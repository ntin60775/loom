# Ownership-aware guard локализации собственных текстовых артефактов

## Когда использовать

Использовать, когда задача затрагивает:
- созданные агентом `.md`-документы, текстовые отчеты, планы, заметки и другие текстовые артефакты;
- создание нового UI или добавление/изменение в UI любых пользовательских надписей, лейблов, кнопок, меню, диалогов, placeholder, helper/error text и других видимых строк;
- явную проверку конкретных документов или UI-файлов по запросу пользователя.

## Цель

Перед завершением изменений убедиться, что собственные пользовательские текстовые артефакты текущей задачи:
- содержат русскоязычную прозу;
- не состоят в основном из англоязычных предложений;
- сохраняют английский только для машинно-значимых литералов, команд, путей, URL, ID и технических терминов;
- в случае UI охватывают все видимые пользовательские тексты созданного или изменённого интерфейса в зоне ответственности текущей задачи;
- не расширяют локализационный scope на чужие или скачанные артефакты без явной просьбы пользователя.

## Базовый порядок

1. Определить scope проверки:
   - если создаётся новый UI-файл или новый UI-экран с пользовательскими строками, считать прохождение guard-а обязательным перед завершением задачи;
   - если нужны только собственные новые документы и добавленные UI-тексты текущей задачи, запускать CLI без путей;
   - если есть сомнение, что default-scope охватит все видимые UI-строки, дополнительно передавать конкретный UI-файл явно;
   - если пользователь явно попросил проверить конкретный файл, каталог, скачанный навык или чужой артефакт, передавать путь явно.
2. Запустить доступную точку входа. Не вызывать голый `check-docs-localization`, пока `command -v check-docs-localization` не подтвердил,
   что такая короткая команда реально есть в текущем `PATH`.

   Предпочтительный порядок:
   - если в репозитории есть скрипт-обёртка, использовать его:

```bash
# Примеры обёрток, которые пользователь может создать в своём репозитории:
bash scripts/check-docs-localization.sh
bash scripts/check-docs-localization.sh README.md docs/roadmap/master-roadmap.md
bash scripts/check-docs-localization.sh src/App.tsx
```

   - если используется установленная глобальная копия навыка, вызывать явную скрипт-обёртку или Python CLI по абсолютному пути:

```bash
~/.agents/skills/owned-text-localization-guard/scripts/check-docs-localization
python3 ~/.agents/skills/owned-text-localization-guard/scripts/markdown_localization_guard.py README.md
```

   - если проверяется исходный пакет этого репозитория, вызывать локальный CLI из репозитория:

```bash
# Внутренний путь в конкретной среде; альтернатива — глобальная копия навыка:
python3 skills-global/owned-text-localization-guard/scripts/markdown_localization_guard.py README.md
```

   - для машиночитаемого вывода и автоматизации отчётов рекомендуется использовать `--format=json` и `--dry-run`:

```bash
python3 ~/.agents/skills/owned-text-localization-guard/scripts/markdown_localization_guard.py --format=json --verbose --dry-run
python3 ~/.agents/skills/owned-text-localization-guard/scripts/markdown_localization_guard.py --format=json src/App.tsx
```

3. Если есть нарушения:
   - создать `SetTodoList` с задачами по исправлению локализации для каждого проблемного файла;
   - исправить англоязычную прозу;
   - оставить английский только там, где он действительно машинно-значим;
   - не трогать чужие артефакты, если их локализация не была явно запрошена;
   - повторно прогнать проверку и отметить todo выполненным.

## CLI-аргументы

| Флаг | Описание | По умолчанию |
|------|----------|--------------|
| `paths` | Файлы или каталоги для явной проверки | — |
| `--root` | Корневой каталог проверки | `cwd` |
| `--allow-phrase` | Дополнительная разрешённая фраза (можно несколько) | — |
| `--ignore-rel-glob` | Glob-шаблон относительно `--root` для игнорирования | — |
| `--max-issues-per-file` | Максимум проблемных строк на файл | `10` |
| `--resolve-relative-to-root` | Разрешать пути относительно `--root` | `False` |
| `--since-minutes` | Для fallback: сколько минут назад считать файлы новыми | `60` |
| `--strict-git` | Падать при недоступности git вместо fallback | `False` |
| `--format` | Формат вывода: `text` или `json` | `text` |
| `--verbose` | Выводить список файлов в scope | `False` |
| `--dry-run` | Показать scope без анализа | `False` |
| `--no-local-config` | Не загружать `.localization-guard.yml` | `False` |

## Конфигурация проекта

В корне проекта можно создать файл `.localization-guard.yml` для кастомизации правил:

```yaml
allowed_phrases:
  - "Docker Compose"
  - "CI/CD"
ignore_rel_globs:
  - "docs/external/**"
doc_extensions:
  - ".md"
  - ".mdx"
ui_extensions:
  - ".jsx"
  - ".tsx"
  - ".vue"
  - ".py"
ui_context_markers:
  - "label"
  - "t("
  - "$t("
max_issues_per_file: 10
```

Если `PyYAML` не установлен, guard выведет понятную ошибку с инструкцией.
Для игнорирования конфига используйте `--no-local-config`.

### Дефолтные значения

**Разрешённые фразы** (`DEFAULT_ALLOWED_PHRASES`):
`README.md`, `AGENTS.md`, `Parent ID`, `markdown-localization-guard`, `owned-text-localization-guard`, `SQLite`, `Markdown`, `JSON`, `YAML`, `XML`, `MCP`, `MVP`, `GitHub`, `OpenAI`, `Anthropic`, `Gemini`, `OpenRouter`, `CLI`, `API`, `HTTP`, `SDK`, `UI`, `UX`, `CSS`, `HTML`, `DOM`, `URL`, `SQL`, `HTTPS`, `REST`, `GraphQL`, `npm`, `yarn`, `webpack`, `babel`, `eslint`, `prettier`, `jest`, `docker`, `kubernetes`, `nginx`, `redis`, `mongodb`, `postgresql`, `firebase`, `aws`, `gcp`, `azure`, `e.g.`, `i.e.`, `etc.`, `vs.`, `Dr.`, `Mr.`, `Mrs.`, `Ms.`.

**Игнорируемые каталоги** (`DEFAULT_IGNORED_TREE_PARTS`):
`.git`, `.venv`, `dist`, `node_modules`.

**Поддерживаемые расширения документов** (`DEFAULT_DOC_EXTENSIONS`):
`.md`, `.mdx`, `.txt`, `.rst`, `.adoc`.

**Поддерживаемые расширения UI** (`DEFAULT_UI_EXTENSIONS`):
`.html`, `.htm`, `.xhtml`, `.jsx`, `.tsx`, `.vue`, `.svelte`, `.js`, `.ts`.

## Коды возврата

| Код | Значение |
|-----|----------|
| `0` | Проверка пройдена, нарушений не найдено. |
| `1` | Ошибка выполнения (неверные аргументы, отсутствие прав доступа, внутренняя ошибка). |
| `2` | Найдены нарушения локализации (exit_code=2 в JSON-ответе). |

## Структура JSON-ответа (`--format=json`)

```json
{
  "exit_code": 0,
  "scope": "default|explicit",
  "targets_checked": 3,
  "targets": ["README.md", "docs/vision.md"],
  "failures": [
    {
      "file": "README.md",
      "issues": ["Строка с англоязычной прозой..."]
    }
  ],
  "message": "Проверка локализации не пройдена:\n\n- README.md\n  * Строка с англоязычной прозой..."

```

## Обработка ошибок

- **PyYAML не установлен**: при наличии `.localization-guard.yml` выводится понятная ошибка с инструкцией по установке.
- **Git недоступен**: автоматический fallback на mtime (файлы, изменённые за `--since-minutes`). С `--strict-git` — ошибка и выход.
- **Нет прав на чтение файла**: файл пропускается с предупреждением в stderr.
- **Лимит файлов**: `--max-files` ограничивает количество проверяемых файлов для защиты от случайного запуска в огромных деревьях.
- **YAML с неизвестными ключами**: выводится предупреждение в stderr, неизвестные ключи игнорируются.

## Нормализация текста перед проверкой

Перед анализом строка проходит нормализацию (`normalize_line`):
- Удаляются YAML frontmatter, inline-код (`` `...` ``), URL.
- Удаляются переменные `${...}`, `{{...}}`, JSX-шаблоны `{...}`.
- Удаляются markdown-чекбоксы (`- [x] `), inline-ссылки и reference-ссылки.
- Удаляются разрешённые фразы (с учётом границ слов для однословных).
- Удаляется пунктуация, остаются только слова.

## Правила интерпретации

- Запуск без путей ownership-aware: он смотрит только на новые файлы целиком и добавленные строки в git diff текущей задачи.
- Если git недоступен или репозиторий не инициализирован, автоматически включается **fallback-режим**: проверяются файлы, изменённые за последние 60 минут (настраивается через `--since-minutes`). В fallback-режиме все подходящие файлы проверяются целиком (`mode='full'`), а не только добавленные строки.
- Это не markdown-only guard: в default-scope входят и пользовательские UI-тексты, а в explicit-scope можно передавать любые поддерживаемые текстовые файлы.
- Для UI contract трактуется широко по пользовательскому смыслу: через guard должны проходить все видимые надписи и тексты, которые агент добавляет или меняет в интерфейсе (веб, мобильный, десктоп, TUI).
- Если UI создаётся с нуля, безопасный default — считать весь файл зоной проверки; если UI меняется точечно, минимумом являются все добавленные агентом пользовательские строки.
- Существующие скачанные навыки, внешние документы и прочие не созданные агентом артефакты не попадают в default-scope сами по себе, **если они закоммичены**. Незакоммиченные (untracked) файлы попадут в git-режиме; в fallback-режиме попадут по mtime.
- Явно переданный путь считается явной просьбой проверить этот файл или каталог независимо от ownership.
- Команды, пути, URL, идентификаторы и названия внешних проектов не считаются нарушением сами по себе.
- Полностью пустой `.md` допустим как осознанный stub-файл и не требует русскоязычной прозы.
- Для новых документов проверяется весь файл (`mode='full'`), а для существующих файлов по умолчанию проверяются только добавленные агентом строки (`mode='added-lines'`).
- В режиме `full` текст очищается от inline-кода, URL и YAML-frontmatter перед анализом. В режиме `added-lines` проверяются сырые добавленные строки без очистки.
- Если новый документ почти полностью англоязычный и не содержит русскоязычной прозы, это считается ошибкой.
- Для UI-файлов проверяются только потенциально пользовательские текстовые фрагменты, а не весь исходный код целиком.
- Code blocks в markdown: блоки без языка или с языком `text`/`plain`/`dialog`/`ui` проверяются как обычный текст; блоки кода (`python`, `bash`, `js` и т.д.) пропускаются.
- i18n-функции (`t('...')`, `$t('...')`, `tr('...')`, `gettext('...')`) детектируются автоматически. Snake_case, camelCase, kebab-case и dot-notation ключи (например, `save_button`, `submitButton`, `submit-button`, `common.save`) считаются машинными идентификаторами и не флагируются.
- ICU-атрибуты (`id='submitForm'`, `defaultMessage='Hello'`) детектируются в JSX/React Intl контексте. `id` считается техническим ключом; `defaultMessage` проверяется как пользовательский текст.
- Маркеры `⟦⟧` используются для директив `AGENTS.md` и распознаются как машинные директивы.
- `--dry-run` показывает scope без анализа; `--strict-git` отключает fallback на mtime; `--resolve-relative-to-root` разрешает пути относительно `--root`; `--max-files` и `--skip-symlinks` описаны в таблице CLI-аргументов выше.
- `.localization-guard.yml` создаётся вручную пользователем проекта; агент не должен автоматически создавать этот файл.

## Итоговый вывод

В финальном отчёте коротко указать:
- что именно проверялось;
- был ли это ownership-aware default-scope, fallback или явный scope по путям;
- прошла ли локализация;
- если были исправления, какие документы пришлось править.

При использовании `--format=json` агент может парсить структурированный вывод для автоматизации отчётов.
