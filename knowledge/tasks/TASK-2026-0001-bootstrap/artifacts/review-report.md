## 2. SDD — основные разделы

### 2.1 Архитектурные инварианты (раздел 2)
✅ **Хорошо.** 11 инвариантов покрывают все ключевые решения. Каждый имеет проверяемый критерий. Инварианты INV-9 (исполнение строго последовательно) и INV-6 (чистые сессии) взаимно усиливают друг друга.

### 2.2 Компонентная модель (раздел 3)
✅ **Хорошо.** 6 компонентов с чёткими границами. Spawner — критический bridging компонент между pi и tmux. Knowledge I/O — чистый data access слой.

### 2.3 Модель данных (раздел 4)
⚠️ **Найдены расхождения:**

| # | Проблема | Где | Серьёзность |
|---|----------|-----|-------------|
| 1 | `PlanStep.title` и `description` — не `Localized`. Оператор видит их в TUI и `/plan-review`. | 4.2 | Низкая (исправимо) |
| 2 | `registry.json` использует `title_ru` вместо `Localized {ru}` по схеме | 8 / registry.json | Средняя — схема и реальность расходятся |
| 3 | `Reviewer auto-select` содержит `.os` — нестандартное расширение для 1С | 6.7 | Низкая (должно быть `.1s`) |

### 2.4 Flow: Plan Mode (раздел 5)
✅ **Хорошо.** System prompt чётко разделяет ответственность: не пиши код, только артефакты. Finalize предлагает чистую сессию — правильное решение для избежания контекстного мусора.

### 2.5 Flow: Agent Mode (раздел 6)
⚠️ **Неоднозначность:**

| # | Проблема | Где | Серьёзность |
|---|----------|-----|-------------|
| 4 | Кто делает git commit? Раздел 6.3 даёт executor tool `git_commit`, раздел 6.5 даёт worker: `git add -A && git commit`. Раздел 6.4 — executor делает «git commit (если надо)». **Решение:** worker делает первичный commit, executor делает amend/дополнительные commits (review.json, merge). Нужно явно прописать. | 6.3 vs 6.5 | Средняя — ambiguity в реализации |

### 2.6 Flow: Onboarding (раздел 7)
✅ **Хорошо.** Pipeline покрывает все сценарии: clean, foreign_system, TCK migration. Разделение между generic onboarding (7.2) и TCK-specific migration (7.4) логично.

⚠️ **Проблема:**

| # | Проблема | Где | Серьёзность |
|---|----------|-----|-------------|
| 5 | TCK migration example показывает статус "active" → "в_работе", но **нет полной таблицы маппинга**: active→?, черновик→?, и т.д. | 7.4.4 | Средняя — миграция недетерминирована без маппинга |

### 2.7 Knowledge Storage (раздел 8)
✅ **Хорошо.** Полная структура с разделением: tasks, modules, project, operations. Дублирование JSON + Markdown оправдано (primary + derivative).

### 2.8 Git Flow (раздел 9)
✅ **Хорошо.** Три типа веток, task-scoped commits, review через git diff. Встроено в процесс, не отдельно.

### 2.9 UI (раздел 10)
✅ **Хорошо.** Фазированный подход: MVP — tmux + базовый widget, фаза 2 — полный TUI. Правильный прагматизм.

### 2.10 Команды extension (раздел 11)
✅ **Хорошо.** 9 команд покрывают все режимы. Разделение на Plan-only и Agent-only предотвращает misuse.

### 2.11 MVP Scope / DU (раздел 13)

### 2.12 Зависимости от TCK (раздел 15)
✅ **Хорошо.** Таблица с чёткими «что берём / что делаем иначе». Это важно для понимания, почему loom — не клон TCK.

### 2.13 Deferred scope (раздел 16)
✅ **Хорошо.** Два слоя отложены (Memory, Scout Retrieval), один исключён (параллельные workers). Обоснования детальны.

## 3. AGENTS.md — ревью

✅ Базовые правила корректны: identity, AI-native формат, task lifecycle, code ownership, git flow.
⚠️ Не хватает явного упоминания команд loom (`/plan`, `/agent`, `/loom-init`) — агент может не знать как войти в plan mode без подсказки. Но это может быть и намеренно (AGENTS.md минимален, команды предоставляет extension).

## 4. task.md — ревью

⚠️ Несколько устаревших формулировок:
- INV-7: «aide = режим работы pi» → должно быть «loom = режим работы pi»
- «Человекочитаемое описание» — человекоцентричное поле, противоречит AI-Native принципу. Но это исторический формат, task.json — канонический

## 5. registry.json — ревью

⚠️ Поле `title_ru` вместо `title: {ru: "..."}`. Не соответствует JSON schema из раздела 4.1 (Localized). Реестр использует плоскую структуру, schema — вложенную. Нужно привести к единообразию.
